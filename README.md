# Fishing Events Bot

Скрипт следит за двумя страницами со списком ивентов (Virtual Fishing / Magic
Fishing) и присылает в Telegram уведомления о событиях с множителем **x5–x10**:

- https://www.fungamescentral.com/vf_lands.php
- https://www.thesecondfarm.com/MagicFishing/BuoysList

Работает как GitHub Actions workflow. У GitHub Actions есть жёсткое
ограничение — расписание (`cron`) не может быть чаще, чем раз в 5 минут.
Поэтому workflow триггерится каждые 5 минут, а внутри самого запуска скрипт
крутится в цикле 5 раз с паузой ~55 секунд — получается проверка примерно
**раз в минуту** (с поправкой на возможные задержки запуска runner'а самим
GitHub, см. раздел "Важные нюансы"). Реально отправляет сообщения только в
окне **10:00–03:00** (по умолчанию таймзона `Europe/Amsterdam`, можно
поменять).

## Как это работает

1. `monitor.py` парсит обе страницы, вытаскивает множитель события (x1…x10)
   для каждого участка.
2. Если сейчас время вне окна 10:00–03:00 — скрипт молча завершается, ничего
   не отправляет.
3. Если время подходящее, для каждого события с множителем 5–10 скрипт
   проверяет `state.json`: если для этого участка уже отправлялось
   уведомление именно с таким множителем — повторно не шлёт. Если событие
   новое или множитель изменился — шлёт сообщение в Telegram и обновляет
   `state.json`.
4. Если событие вообще пропадает из списка (закончилось), запись о нём
   удаляется из `state.json`, чтобы при повторном появлении снова прислать
   уведомление.
5. Workflow коммитит обновлённый `state.json` обратно в репозиторий, чтобы
   состояние сохранялось между запусками.

## Настройка (5 шагов)

### 1. Создать Telegram-бота
Напишите [@BotFather](https://t.me/BotFather) в Telegram → `/newbot` →
следуйте инструкциям → скопируйте **токен** (выглядит как
`123456789:AAExxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx`).

### 2. Узнать свой chat_id
Проще всего написать [@userinfobot](https://t.me/userinfobot) — он пришлёт
ваш `chat_id`. Либо: напишите что-нибудь своему новому боту, затем откройте
`https://api.telegram.org/bot<ТОКЕН>/getUpdates` в браузере — там будет поле
`"chat":{"id": ...}`.

### 3. Создать репозиторий на GitHub
Создайте новый (можно приватный) репозиторий и залейте туда все файлы из
этого проекта, сохраняя структуру папок (важно, чтобы workflow остался в
`.github/workflows/check_events.yml`).

### 4. Добавить секреты
В репозитории: **Settings → Secrets and variables → Actions → New repository
secret**, добавьте два секрета:

- `TELEGRAM_BOT_TOKEN` — токен бота из шага 1
- `TELEGRAM_CHAT_ID` — chat_id из шага 2

### 5. Включить Actions и проверить
Перейдите во вкладку **Actions**, откройте workflow "Check Fishing Events" →
**Run workflow** (кнопка справа), чтобы запустить вручную и проверить, что
всё работает. Дальше он будет запускаться сам каждые 10 минут.

## Настройка под себя

Все параметры задаются переменными окружения в файле
`.github/workflows/check_events.yml`:

| Переменная       | По умолчанию         | Описание                                    |
|------------------|-----------------------|----------------------------------------------|
| `TIMEZONE`       | `Europe/Amsterdam`    | Таймзона для окна уведомлений                |
| `WINDOW_START`   | `10:00`               | Начало окна (локальное время)                |
| `WINDOW_END`     | `03:00`               | Конец окна (может быть "через полночь")      |
| `MIN_MULTIPLIER` | `5`                   | Минимальный множитель для уведомления        |
| `MAX_MULTIPLIER` | `10`                  | Максимальный множитель для уведомления       |

## Важные нюансы

- **Чаще, чем раз в 5 минут, GitHub cron в принципе не умеет** — это
  жёсткое ограничение платформы, обходится только "трюком" с циклом внутри
  одного запуска (как сделано здесь).
- **GitHub Actions cron не гарантирует точность** — при высокой нагрузке на
  GitHub сам запуск может задержаться на 10-30 минут (в редких случаях
  больше), это не зависит от того, что написано в расписании.
- **Лимит минут для приватных репозиториев.** Каждый запуск занимает ~5
  минут (из-за цикла с sleep). При триггере раз в 5 минут это ~1440 минут
  Actions в сутки — бесплатный лимит для приватных репо (2000 мин/месяц)
  закончится за пару дней. Варианты:
  - Сделать репозиторий **публичным** — для публичных репозиториев минуты
    Actions бесплатны без лимита.
  - Оставить приватным, но снизить частоту (например, проверять раз в
    5 минут без внутреннего цикла — тогда расход минут будет в разы
    меньше). Уберите цикл `for i in 1 2 3 4 5` в workflow-файле и оставьте
    просто `python monitor.py`.
- Если 60 дней подряд не будет активности в репозитории, GitHub может
  автоматически отключить scheduled workflow — в этом случае зайдите во
  вкладку Actions и включите его заново.
- Структура таблиц на сайтах может измениться — если парсинг сломается,
  в логах запуска (вкладка Actions → последний run) будет видно ошибку.

## Локальный запуск (для теста)

```bash
pip install -r requirements.txt
export TELEGRAM_BOT_TOKEN="..."
export TELEGRAM_CHAT_ID="..."
python monitor.py
```

Чтобы протестировать вне окна 10:00–03:00, можно временно расширить окно,
например `export WINDOW_START=00:00 WINDOW_END=23:59`.
